## Added

### Phase 9.6 — runbook Patroni 자동 cutover 시나리오 (§11)

`docs/operations/multi-region-failover-runbook.md` §11 신설.

#### 11.1 자동 단계 timeline

```
T+0:00  PG primary pod 죽음
T+0:05  Patroni leader lease 만료 감지 (etcd TTL 30s)
T+0:10  Patroni standby × 2가 promote 시도 → etcd quorum 확인 → 1개 promote
T+0:15  새 PG primary가 wal stream 수용 시작
T+0:20  rosshield-server pod 3개가 Patroni REST polling → 새 leader 식별
T+0:21  application-level RoleProvider epoch 증가 → write API 수용 시작
```

**RTO**: ~30초.

#### 11.2 운영자 검증 5 step

1. PagerDuty 알림 + Slack 통지 (acked-by + time)
2. `patronictl list` (1 Leader + 2 Replica + TL 증가 확인)
3. `kubectl exec ... curl localhost:8080/healthz | jq '.ha'` 모든 pod RoleProvider 추종 확인
4. write API `POST /api/v1/audit/checkpoint` 200 확인
5. Customer 통지 (impact ~30초)

#### 11.3 false positive `patronictl pause`

network partition · etcd flapping · PG load 등 의심 시 자동 failover 즉시 정지 + 원인 분석 + `patronictl resume`.

#### 11.4 manual `patronictl failover --force`

Patroni 자동 cutover 실패 시 강제 promote 절차 (etcd quorum 우회).

#### 11.5 E25 fallback (D-AF-4)

Patroni 또는 etcd 자체 장애 시 `kubectl set env ROSSHIELD_HA_RP=e25` + `kubectl rollout restart`로 PG advisory lock 기반 E25 fallback.

#### 11.6 RTO 비교

| 단계 | E25 manual (§3) | Patroni 자동 (§11) |
|---|---|---|
| 감지 | T+0:00 ~ T+2:00 | T+0:00 ~ T+0:10 |
| PG promote | T+2:30 (수동) | T+0:10 (자동) |
| Application restart | T+3:00 (수동) | T+0:20 (자동, polling) |
| Client 검증 | T+4:00 (수동) | T+0:21 (자동) |
| **총 RTO** | **≤ 5분** | **≤ 30초** |
