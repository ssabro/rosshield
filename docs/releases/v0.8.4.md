## TL;DR

**E35-refresh redesign + carryover 마감 patch** — v0.8.3까지 누적되어 있던 snap `post-refresh` hook 기반 healthz polling 설계가 snap lifecycle 표준과 architectural mismatch(post-refresh는 services start **전** 호출됨)임을 확정. hook 책임을 분리(post-refresh = 무결성 sanity, check-health = daemon healthz polling)하고 CI snap-smoke의 refresh round-trip step에서 `continue-on-error: true`를 제거하여 E35-refresh carryover 마감. 자동 rollback 범위는 catastrophic case(binary corruption + configure schema)로 축소되고, daemon unhealthy 대응은 운영자/Prometheus alert로 위임. Breaking 0, 회귀 0.

## Added

### snap `check-health` hook 신규 (`3b8e36c`)

snapd가 services start 후 호출 + 주기적(약 5분) 재호출하는 표준 hook. daemon healthz polling 책임을 전담합니다.

- healthz polling max 30s cap (snapd docs "fast + idempotent" 권장 일관).
- healthy → `snapctl set-health okay`, unhealthy → `snapctl set-health waiting` (transient 의미).
- exit 0 default — snapd는 hook exit status 무시, `set-health` 호출 결과만 활용.
- 자동 revert 트리거 안 함 — 운영자 또는 Prometheus alert로 위임.

### snap-smoke CI `snap health` polling step 2개 (`02a8f6c`)

- `snap health — check-health hook status` (첫 install 직후): max 30s polling으로 `snap health rosshield` okay 도달 확인. 빈 status는 "no health concerns" 의미 → okay와 동등 처리.
- `snap health — check-health after refresh` (refresh round-trip 후): 새 revision의 daemon이 check-health okay까지 도달 확인.

### 설계 doc `e35-refresh-rollback-redesign.md` (`3b8e36c`)

9 섹션 약 270줄. snap hook lifecycle fact-check 표 + 옵션 A·B·C·D 비교 + Stage 1~4 분해 + D-E35R-1·2·3 결정 기록. 자동 rollback과 healthz polling을 동시에 만족하는 snap hook은 표준에 존재하지 않는다는 사실을 명시.

### 운영자 docs `snap-deployment.md §7` 전면 rewrite (`3419191`)

"자동 rollback (E35)" → "health 모니터링 (E35 redesign)"으로 제목 변경 + 7개 sub-section 구성:

- §7.1 hook 책임 분리 표 (post-refresh / check-health / configure × 호출 시점·책임·exit 1 동작).
- §7.2 자동 rollback 범위 (축소됨) — catastrophic case만 cover 명시.
- §7.3 health 모니터링 — `snap health rosshield` 명령 + Prometheus alert rule 예제.
- §7.4 healthz 폴링 정책 조정 — configure hook 검증용 vs. check-health max 30s cap 분리.
- §7.5 broken refresh 대응 절차 (5단계: snap health → curl → logs → revert → hold).
- §7.6 채널 정책 — edge / candidate 3~7일 burn-in / stable 수동 승급, 자동 rollback 부재의 1차 방어선.
- §7.7 OTA round-trip 검증 — CI 자동 검증 + 수동 검증 절차에 snap health okay 도달 단계 추가.

## Changed

### snap `post-refresh` hook 단순화 (`3b8e36c`)

기존 healthz polling 코드 완전 제거 (architectural mismatch — post-refresh는 services start 전 호출). 신규 책임:

- binary 실행 가능 + 비어있지 않음 + healthz-url prefix sanity check만 수행.
- exit 1은 binary corruption 등 catastrophic case만 — snapd 자동 revert 트리거 보존.
- 정상 시 logger로 sanity OK 출력 + exit 0.

### snap-smoke refresh round-trip step `continue-on-error` 제거 (`02a8f6c`)

`snap refresh — same-snap re-install (round-trip)`:

- **`continue-on-error: true` 제거** — E35-refresh carryover 마감.
- step 이름 단순화("round-trip simulation" → "round-trip", carryover 코멘트 제거).
- daemon restart 후 healthz listen 준비까지 30s grace polling 추가 (단발 jq | exit 1 패턴 → loop polling).
- 정상 응답 시 step PASS, 30s 내 응답 못 받으면 명시적 fail.

기존 step 2개 제거 — post-refresh hook이 healthz polling 안 하므로 무의미:

- `extend healthz-timeout for refresh round-trip` 제거 (timeout 확장 무의미).
- `post-refresh hook — healthz polling on live snap` 제거 (직접 invocation 무의미).

### `snap/snapcraft.yaml` hooks 섹션 (`3b8e36c`)

- `post-refresh.plugs`는 network 유지 (logger + 추후 무결성 확장 cover).
- `check-health` hook 신규 등록 + `plugs=[network]` (curl localhost healthz).

### `snap-deployment.md §9` 한계 갱신 (`3419191`)

- "A/B OTA + 자동 rollback" 항목 — 1차 설계의 architectural mismatch 발견 + 2026-05-21 redesign + 자동 rollback 범위 축소 사실 명시.
- 신규 한계 2 추가:
  - post-refresh hook은 daemon health 검증 불가 (snap lifecycle 표준 제약).
  - snap store channel staged rollout — docs 권장(§7.6)만 cover, 실 등록 + CI release pipeline 분리는 별 epic.

## 한계 (자동 rollback 범위 축소)

E35-refresh redesign의 결과로 자동 rollback 가정이 다음과 같이 좁아졌습니다:

- **자동 rollback이 트리거되는 경우**: snap binary corruption / configure hook schema 위반 (post-refresh exit 1 또는 configure exit 1).
- **자동 rollback이 트리거되지 않는 경우**: daemon 시작 후 healthz 응답 불가, daemon panic crash loop, downstream dependency 실패 등 일반적 unhealthy 상태.
- **대안 1차 방어선** (docs §7.6): snap store channel staged rollout — edge → candidate 3~7일 burn-in → stable 수동 승급.
- **대안 2차 방어선** (docs §7.3·§7.5): `snap health rosshield` Prometheus alert + broken refresh 대응 절차(5단계).

운영자는 `docs/operations/snap-deployment.md §7`을 정독한 뒤 customer 환경에 채널 정책 + alert rule을 적용해야 합니다.
