### audit-verify CLI에 cosign verify 통합 (`7e19a9f`)

v0.6.8 한계 항목 "rosshield-audit-verify CLI cosign verify 통합" 해소. 외부 감사인이 segment 무결성 + cosign keyless 서명을 단일 도구로 검증할 수 있게 됨.

#### rotation single mode 신규 flag

```bash
rosshield-audit-verify rotation \
    --archive-uri file:///path/to/seg-000005.tar.gz \
    --cosign-bundle /path/to/seg-000005.cosign.bundle \
    --cosign-identity ci@example.com \
    --cosign-oidc-issuer https://accounts.google.com
```

5 flag: `--cosign-bundle`, `--cosign-identity`, `--cosign-oidc-issuer`, `--cosign-binary`, `--cosign-rekor-url`.

#### chain mode

```bash
rosshield-audit-verify rotation chain \
    --backend file:///path/to/audit-archives/<tenant>/ \
    --from-segment 1 --to-segment 10 \
    --cosign-bundle-dir /path/to/bundles/ \
    --cosign-identity ci@example.com
```

`--cosign-bundle-dir`로 각 segment 옆 `seg-NNNNNN.cosign.bundle` 자동 검색.

#### 결정

- `cosign verify-blob` 외부 CLI exec 채택 (sigstore-go SDK 임베드 회피)
- 이유: verify CLI는 E30 + R30-4 취지로 stdlib + crypto/ed25519 + 도메인 hash만 의존. dep 추가 시 외부 검증 자산 사이즈/감사 영향
- `runCosignVerify` function var pattern으로 test 격리 (외부 cosign binary 의존 0)

#### Test

5 신규: skip / verify PASS / runner error / bundle 부재 / chain bundle-dir 자동 검색.

### S3 lifecycle 자동 적용 (`42d44cc`)

v0.6.8 한계 항목 "S3 lifecycle 자동 적용 — Glacier transition 등" 해소.

#### S3Config 확장

```go
type S3Config struct {
    // ... 기존 필드 ...
    LifecycleEnabled     bool
    LifecycleTransitions []S3Transition  // STANDARD_IA → GLACIER → DEEP_ARCHIVE 단계
    LifecycleExpireDays  int32           // 0 = 영구
}

type S3Transition struct {
    Days         int32
    StorageClass string
}
```

#### ApplyLifecyclePolicy

- Rule ID = "rosshield-rotation" 고정 (운영자가 AWS console에서 식별·삭제)
- Filter.Prefix = cfg.Prefix (다른 application 객체 영향 0)
- idempotent (PutBucketLifecycleConfiguration이 덮어쓰기)
- LifecycleEnabled=true 시 NewS3Backend 자동 호출 (ErrLifecycleEmpty graceful skip)

#### 환경 적용 예시

```bash
export ROSSHIELD_AUDIT_COLD_BACKEND=s3
export ROSSHIELD_AUDIT_S3_LIFECYCLE_ENABLED=true
# (transitions/expire는 bootstrap config 후속 — 본 round는 SDK API + customer 자체 설정 위주)
```

#### Test

6 신규: empty config err / transitions only / expiration only / new auto applies / propagates error / respects ctx cancel.

### Multi-region HA Stage 3 후속 — publication tables sync + slot cleanup (`905fbf8`)

v0.6.8 한계 항목 "publication tables 변경 자동 동기화"·"slot 자동 cleanup" 해소.

#### publication tables 자동 동기화

`syncPublicationTables(ctx, exec, spec)` 신규 — 기존 publication이 존재할 때 spec.Tables와 `pg_publication_tables` diff → ADD/DROP TABLE 자동 발행. `ensurePublication`의 exists 경로에서 자동 호출.

- AllTables=true → no-op (FOR ALL TABLES는 자동 동기)
- AllTables=false → 누락은 ADD, 잉여는 DROP. `pg_publication_tables` 표준 카탈로그 활용 (PG 13+)
- `diffTables` sort 후 반환 → 결정론·테스트 친화

#### dead slot cleanup

```go
func CleanupInactiveSlots(ctx, exec, opts) (removed []string, err error)

type CleanupInactiveSlotsOptions struct {
    MinInactiveAge time.Duration  // default 24h
    DryRun         bool
    SlotPrefix     string          // 안전 prefix 강제 (e.g., "rosshield_")
    Logger         *slog.Logger
}
```

- `pg_replication_slots`에서 `active=false` + N분 이상 stale slot 조회
- 이중 prefix 가드: PG SQL `LIKE` 필터 + client측 `strings.HasPrefix` 재검증
- `pg_drop_replication_slot($1)` bind parameter — SQL injection 차단
- DryRun=true면 SELECT만 + return list

#### Executor 확장

`Executor.QueryStrings(ctx, sql, args) ([]string, error)` 추가 — 단일 컬럼 string list 조회. pgxpool wrap.

#### Test

13 신규: publication sync (AllTables / add / drop / 양쪽 diff) + slot cleanup (drop / DryRun / prefix mismatch / 빈 목록) + helper 단위.

### Web bundle code-splitting (`2c8c8e9`)

v0.6.8 한계 항목 "Bundle size 800KB warning" 해소.

#### `web/vite.config.ts` build.rollupOptions.output.manualChunks 결선

| chunk | raw | gzip |
|---|---|---|
| react-vendor | 194 KB | 60 KB |
| radix-vendor | 92 KB | 27 KB |
| form-vendor | 89 KB | 24 KB |
| ui-vendor | 84 KB | 22 KB |
| router-vendor | 78 KB | 25 KB |
| query-vendor | 56 KB | 17 KB |
| vendor (기타) | 50 KB | 18 KB |
| state-vendor | 2.7 KB | 1.3 KB |
| **index (main + 도메인)** | **239 KB** | **59 KB** |

- 단일 main chunk 835KB → 239KB (수치적으로 ~71% 감소)
- 8 vendor chunk 분리 → browser parallel fetch + vendor 업데이트 시 cache invalidation 범위 최소화
- 500KB chunk size warning **0**
- `node_modules` path filter — `virtual:pwa-register` 등 가상 모듈을 vendor로 끌어가지 않음 (vitest mock 패턴 보존)
