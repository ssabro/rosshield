## Added

### Multi-region HA Stage 3 후속 — slot cleanup cron 자동 운영 (`742e0c1`)

v0.6.9의 `setup.CleanupInactiveSlots` helper가 코드만 존재하던 상태 → 정기 cron job으로 자동 실행되도록 wiring 완성.

#### 신규 패키지 `internal/platform/scheduler/replicationcleanupjob/`

`rotationjob` 패턴 일관:
- `Register(sch, deps, jobID, spec)` — spec="" → no-op, **SlotPrefix 빈 값 → fail-fast** (다른 application slot 실수 drop 방지)
- `RunOnce(ctx, deps)` — `setup.CleanupInactiveSlots` wrap + 결과 logging
- 12 단위 test: arg 검증 + DryRun skip + drop SQL 검증 + query/drop error 전파

#### bootstrap 결선

primary PG + replication enabled + schedule != "" 조합에서만 자동 등록:
- HA 활성 시 cronsched RoleProvider gate가 follower tick 자동 skip → leader 단일 인스턴스 보장
- sqlite 또는 standby 환경은 silent skip (warn log)

#### CLI flag 4 + env 4

```bash
--replication-slot-cleanup-schedule '@every 24h'  # cron spec
--replication-slot-cleanup-prefix 'rosshield_'    # 안전 가드
--replication-slot-cleanup-min-inactive-age 24h
--replication-slot-cleanup-dry-run                # 후보만 logging
```

env: `ROSSHIELD_REPLICATION_SLOT_CLEANUP_{SCHEDULE,PREFIX,DRY_RUN}`.

### MinIO Content-MD5 middleware + 통합 검증 (`810171a`)

MinIO·일부 S3 호환 storage는 PutBucketLifecycleConfiguration에 RFC 1864 Content-MD5 헤더를 strict 요구 — AWS SDK v2의 modern ChecksumAlgorithm 모델은 미지원. legacy 호환 middleware 신규.

#### addLegacyContentMD5Middleware

smithy-go finalize middleware:
- request body의 MD5를 base64 인코딩 → Content-MD5 헤더 자동 주입
- `req.SetStream(bytes.NewReader(data))` clone + `in.Request` 갱신 (smithy-go API는 clone 반환)
- PutBucketLifecycleConfiguration 호출에만 한정 적용 (다른 S3 호출 영향 0)
- AWS 본가도 Content-MD5 + ChecksumAlgorithm 양쪽 허용 — 운영 영향 0

#### MinIO integration test 분리

`MinIOExistsAndLifecycle` → `MinIOExists` + `MinIOLifecycle` 2 test:
- MinIOExists: HEAD round-trip
- **MinIOLifecycle: NewS3Backend LifecycleEnabled 자동 적용 + 명시 재호출 idempotency** (middleware 정상 동작 시에만 PASS)

## Fixed

### S3 lifecycle Content-MD5 (1차 시도, `666682d`)

PutBucketLifecycleConfigurationInput에 `ChecksumAlgorithm: ChecksumAlgorithmSha256` 추가. AWS 본가는 정상이지만 MinIO는 인식 못함 — 위 middleware 도입으로 최종 해소.

### MinIO integration test lifecycle 분리 (`201c71d`)

middleware 도입 전 임시 조치 — TestS3Backend_MinIOExistsAndLifecycle을 축소. middleware 도입 후 다시 복원(`810171a`).

### TestRunCancelSkipsRemainingButWaitsInFlight flaky 안정화 (`c3ec3f1`)

CI runner 부하 시 timing 의존성 fail:
- exec duration 100ms → 500ms
- cancel sleep 50ms → 100ms
- workerLimit=2 + sufficient cancel window 보장

### go.mod tidy drift (`01fafa1`)

`aws-sdk-go-v2/credentials` — MinIO integration test가 `NewStaticCredentialsProvider`를 직접 사용하면서 indirect → direct dep 승격.

## Migration Notes

### slot cleanup 활성화 (옵션)

```bash
# Primary region only
export ROSSHIELD_REPLICATION_ENABLED=1
export ROSSHIELD_REPLICATION_ROLE=primary
export ROSSHIELD_REPLICATION_SLOT_CLEANUP_SCHEDULE='@every 24h'
export ROSSHIELD_REPLICATION_SLOT_CLEANUP_PREFIX=rosshield_

# Dry-run으로 후보 확인 후 활성
export ROSSHIELD_REPLICATION_SLOT_CLEANUP_DRY_RUN=1
```

PG `pg_replication_slots`에서 `slot_name LIKE 'rosshield_%'` + `active=false` + N분 이상 stale slot이 cleanup 대상. SlotPrefix 명시 안 하면 fail-fast로 부팅 차단 (안전 가드).

### S3 lifecycle 활성화 (옵션, enterprise 빌드)

```bash
go build -tags rosshield_enterprise -o bin/rosshield-server ./cmd/rosshield-server

export ROSSHIELD_AUDIT_COLD_BACKEND=s3
export ROSSHIELD_AUDIT_S3_BUCKET=acme-audit
export ROSSHIELD_AUDIT_S3_REGION=us-west-2
export ROSSHIELD_AUDIT_S3_LIFECYCLE_ENABLED=1
export ROSSHIELD_AUDIT_S3_LIFECYCLE_TRANSITION_IA_DAYS=365
export ROSSHIELD_AUDIT_S3_LIFECYCLE_TRANSITION_GLACIER_DAYS=1825
export ROSSHIELD_AUDIT_S3_LIFECYCLE_EXPIRE_DAYS=2555
```

MinIO·Wasabi·Backblaze B2 등 S3 호환 storage:
```bash
export ROSSHIELD_AUDIT_S3_ENDPOINT=https://s3.minio.local:9000
export ROSSHIELD_AUDIT_S3_FORCE_PATH_STYLE=1
```

GLACIER·DEEP_ARCHIVE transition은 MinIO에서 silent ignore 가능 — storage 측 정책 위임.
