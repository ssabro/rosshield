## Fixed

### Snap Build snapcraft App command validation (`7bb7b9b`)

#### Root cause

v0.8.2 Snap Build CI run에서:
```
Value error, App commands must consist of only alphanumeric characters, spaces,
and the following characters: / . _ # : $ -
input_value='bin/rosshield-server --addr=127.0.0.1:8080'
```

snapcraft App commands는 `=` 문자 거부. `--addr=127.0.0.1:8080`을 command line에 직접 명시 불가.

#### Fix

server-side env 인식 패턴으로 우회 — wrapper script 도입 없이 단순.

**main.go**:
```go
defaultAddr := "127.0.0.1:0"
if envAddr := strings.TrimSpace(os.Getenv("ROSSHIELD_ADDR")); envAddr != "" {
    defaultAddr = envAddr
}
addr := flag.String("addr", defaultAddr, "bind address (also ROSSHIELD_ADDR env)")
```

CLI `--addr=...` 플래그는 env보다 우선 — Go flag 표준 패턴.

**snapcraft.yaml**:
```yaml
apps:
  server:
    command: bin/rosshield-server     # `=` 제거
    environment:
      ROSSHIELD_ADDR: 127.0.0.1:8080  # env로 default 재정의
```

#### 부수 이점 — Kubernetes·Docker·systemd 동시 cover

`ROSSHIELD_ADDR` env 패턴은 snap 외 환경에서도 유용:
- Kubernetes Deployment에서 ConfigMap·env로 주입
- Docker `-e ROSSHIELD_ADDR=...`
- systemd `Environment=ROSSHIELD_ADDR=...`

운영자가 Lodestar binary를 다양한 deploy target에서 사용 시 일관 인터페이스.

## Migration Notes

### snap customer

영향 0 (`snap/snapcraft.yaml`이 자동으로 `ROSSHIELD_ADDR=127.0.0.1:8080` 주입).

### Kubernetes·Docker·systemd customer

신규 옵션 — env로 bind address 재정의 가능:
```bash
docker run -e ROSSHIELD_ADDR=0.0.0.0:8080 lodestar/rosshield-server
```

```yaml
# kubernetes deployment
env:
  - name: ROSSHIELD_ADDR
    value: "0.0.0.0:8080"
```

```ini
# systemd service
Environment="ROSSHIELD_ADDR=127.0.0.1:8080"
```

### 빌드

```bash
git fetch --tags && git checkout v0.8.3
cd web && pnpm install --frozen-lockfile && pnpm build
cd .. && go build -o bin/rosshield-server ./cmd/rosshield-server
```
