## 2.2 코어는 하나, 배포는 세 형태

### 공통 컴포넌트

| 컴포넌트 | 설명 |
|---|---|
| **Core Server** | 도메인 + 플랫폼 서비스 + HTTP/WS API. 단일 바이너리 또는 OCI 이미지. |
| **Web UI** | React 19 + shadcn/ui. 정적 자산 번들. Core가 `/` 하위에 서빙. |
| **Desktop Shell** | Tauri 기반 얇은 네이티브 껍질. Core를 자식 프로세스로 띄우고 localhost:0에 바인딩, WebView로 UI 로드. |
| **CLI** | 같은 HTTP API의 커맨드 클라이언트. |
| **Core Pack** | 기본 벤치마크·매핑·리포트 템플릿을 담은 **서명된 팩**. 제품에 번들되지만 독립 업데이트 가능. |
| **Pack Updater** | 온라인/오프라인 모드. 팩 서명 검증·롤백 지원. |

### 배포 타깃 1: 데스크톱

```
┌─────────────────────────────┐
│  Tauri Desktop Shell        │
│  ┌───────────────────────┐  │
│  │  Web UI (WebView)     │  │
│  └──────────┬────────────┘  │
│             │ localhost:xxxx│
│  ┌──────────▼────────────┐  │
│  │  Core Server (child)  │  │
│  │   + SQLite (file)     │  │
│  │   + SSH Pool          │  │
│  └───────────────────────┘  │
└─────────────────────────────┘
         사용자 PC
```

- **타깃 사용자**: 개인·스타트업·현장 엔지니어
- **데이터**: 로컬 SQLite 파일 (`~/.fleetguard/data.db`)
- **테넌시**: 기본 테넌트 1개
- **업데이트**: 앱 업데이트 + 팩 업데이트 분리
- **SKU**: Free / Pro
- **인증**: 로컬 OS 사용자 계정 (또는 단일 PIN)

### 배포 타깃 2: 온프렘 서버

```
┌──────────────────────────────────────────────┐
│  Kubernetes 또는 Docker Compose              │
│  ┌──────────────┐  ┌──────────────┐          │
│  │ Core Server  │──│  PostgreSQL  │          │
│  │   (N replicas)│  │              │          │
│  └──────┬───────┘  └──────────────┘          │
│         │          ┌──────────────┐          │
│         ├──────────│  Blob Store  │          │
│         │          │  (MinIO)     │          │
│         │          └──────────────┘          │
│         │          ┌──────────────┐          │
│         └──────────│   Ollama     │(옵션)    │
│                    └──────────────┘          │
│         │                                     │
│         ▼ HTTPS                               │
│     사용자 브라우저                            │
└──────────────────────────────────────────────┘
```

- **타깃 사용자**: 중대형 기업 보안팀·운영팀
- **데이터**: PostgreSQL + MinIO(또는 S3 호환 스토리지)
- **테넌시**: 다중 테넌트 (조직별 격리)
- **인증**: SSO(SAML/OIDC), 로컬 계정, API Key
- **업데이트**: Helm 차트 or compose 파일 교체, 팩 업데이트 별도
- **SKU**: Enterprise
- **네트워크**: 로봇 SSH ←→ Core(내부망). 사용자 ←→ Core(HTTPS). 외부 인터넷 선택적.

### 배포 타깃 3: 어플라이언스 이미지

```
┌──────────────────────────────────────────────┐
│  Immutable OS Image (Ubuntu Core / OSTree)   │
│  ┌─────────────────────────────────────────┐ │
│  │  System Layer (read-only)               │ │
│  │   · OS + 필수 패키지 + 보안 업데이트      │ │
│  └─────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────┐ │
│  │  Application Layer (OCI containers)     │ │
│  │   · Core Server                         │ │
│  │   · PostgreSQL (임베디드 or 외장)        │ │
│  │   · Web UI (Core가 서빙)                 │ │
│  │   · Pack Mirror                         │ │
│  │   · Optional: Ollama                    │ │
│  └─────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────┐ │
│  │  Data Layer (writable, encrypted)       │ │
│  │   · /var/lib/fleetguard                 │ │
│  └─────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
   표준 미니 PC 하드웨어 (NUC/OptiPlex 등)
```

- **타깃 사용자**: 방산·공항·병원·원전·중요 제조업, 에어갭 환경
- **OS**: Ubuntu Core 또는 Fedora IoT / OSTree 기반 immutable
- **원자적 업데이트**: OS·컨테이너 이미지·팩을 각각 서명 검증 후 A/B 롤백 가능 슬롯에 적용
- **물리적 보안**: TPM 기반 디스크 암호화, Secure Boot, 콘솔 접근 제한
- **네트워크**: 로봇 네트워크에만 연결. 관리자는 별도 VLAN 또는 직결.
- **SKU**: Appliance / Critical Infra
- **하드웨어**: **우리가 만들지 않음**. 이미지 + 레퍼런스 디자인 + 파트너 채널.
- **에어갭 업데이트**: USB 서명 번들 → 내부 mirror → OTA.

## 2.3 세 배포가 "같은 바이너리"라는 의미

모든 배포 형태는 동일한 Core 바이너리/OCI 이미지를 사용합니다. 차이는:

| 축 | 데스크톱 | 온프렘 | 어플라이언스 |
|---|---|---|---|
| 저장소 | SQLite 파일 | PostgreSQL | PostgreSQL |
| 바인딩 | `127.0.0.1:auto` | `0.0.0.0:443` | `0.0.0.0:443` |
| 인증 | 로컬 | SSO + 로컬 + API Key | SSO + 로컬 + API Key |
| 테넌시 | 기본 1개 | 다중 | 다중 또는 1개(계약별) |
| 셸 | Tauri | 브라우저 | 브라우저 or kiosk WebView |
| 업데이트 채널 | 앱 스토어 or direct | Helm/Compose | A/B OTA |

**구성**은 `config.yaml` (또는 환경변수)로 런타임 분기. 같은 이미지를 다르게 기동.

## 2.4 네트워크 경로

### 인바운드 (Core → 외부)

- **로봇 SSH** (포트 22): 플릿 감사용
- **LLM provider** (옵션): OpenAI/Anthropic/Ollama 엔드포인트
- **Pack Mirror** (옵션, 온라인 모드): 컨텐츠 업데이트 확인·다운로드
- **텔레메트리** (옵션, 기본 off): 제품 사용 통계

모두 **설정에서 개별 차단 가능**. 에어갭 프로필에서는 pack mirror와 telemetry가 강제 off.

### 아웃바운드 (외부 → Core)

- **사용자 브라우저** (HTTPS)
- **CLI 클라이언트** (HTTPS)
- **로봇 Agent** (선택적·옵션): 향후 pull 모드 추가 시 필요

## 2.5 데이터 흐름 (대표 시나리오)

### 시나리오 A: 주기 스캔

```
Scheduler (Platform)
   │ (01:00 UTC 트리거)
   ▼
scan 도메인: createSession → enqueue checks
   │
   ▼
SSH Pool: 각 체크 병렬 실행 → stdout/stderr 수집
   │
   ▼
evidence 도메인: Evidence 저장 (해시 기반 중복제거)
   │
   ▼
scan 도메인: 각 check evaluation → Result
   │
   ├─▶ insight: drift·anomaly·root-cause 계산
   ├─▶ compliance: 프레임워크별 점수 갱신
   ├─▶ audit: 감사 로그 append (해시 체인)
   ▼
reporting 도메인: 세션 요약 Markdown + PDF 생성, 서명
   │
   ▼
Event Bus: ScanCompleted 이벤트 → 구독자(알림·UI 업데이트·Webhook)
```

### 시나리오 B: 온보딩

```
사용자가 "로봇 추가" 클릭
   │
   ▼
onboarding: SSH 진단 실행 → OS/ROS2/Docker 버전 수집
   │
   ▼
benchmark: 감지된 환경에 맞는 vendor 팩 추천
   │
   ▼
onboarding: 추천 baseline/criticality 영속화
   │
   ▼
robot: Robot 엔터티 생성 + fleet에 편입
   │
   ▼
scan: 첫 감사 세션 자동 실행
```

## 2.7 업데이트 모델

### 제품 업데이트 vs 컨텐츠 업데이트

두 축은 **완전히 분리된 생명주기**.

| 유형 | 예시 | 주기 | 배포 경로 |
|---|---|---|---|
| **제품 코드** | Core 서버·UI·CLI | 6-8주 | 앱 스토어 / Helm / OTA 이미지 |
| **벤치마크 팩** | CIS Ubuntu 24.04 업데이트 | 월간·규제 이벤트 기반 | Pack Mirror / 오프라인 번들 |
| **컴플라이언스 매핑 팩** | ISMS-P 개정 반영 | 분기 | Pack Mirror / 오프라인 번들 |
| **리포트 템플릿 팩** | PDF·HTML 서식 | 필요 시 | Pack Mirror / 오프라인 번들 |

구독이 만료되면 **제품은 계속 돌지만 새 팩을 받지 못함** — 가치 감소는 점진적(고객 이탈 방지).

## 2.8 배포 의사결정 가이드 (고객용)

| 고객 상황 | 권장 SKU |
|---|---|
| 1인·소규모, PoC 단계 | 데스크톱 Free → Pro |
| 팀 공유·내부 SSO 필요 | 온프렘 서버 |
| 외부 컴플라이언스 인증 필요, 인터넷 연결 제약 있음 | 온프렘 서버 (에어갭 모드) |
| 법적 에어갭 의무, 물리적 격리 | 어플라이언스 이미지 + 파트너 박스 |
| 로봇 제조사 출하 번들 | OEM 라이선스(데스크톱 또는 임베디드) |

## 2.9 미해결 결정 사항

이 문서 범위에서 확정해야 할 설계 결정:

1. **데스크톱 셸**: Tauri vs Electron vs .NET MAUI — `11-tech-stack-and-roadmap.md`에서 상세 논의.
2. **어플라이언스 OS**: Ubuntu Core(snap) vs OSTree(Fedora IoT) vs 자체 minimal.
3. **Blob Store**: MinIO 포함 번들 vs 고객 제공(S3 호환 외부) 권장.
4. **CLI 배포**: 단일 바이너리 다운로드 vs npm/pip 패키지 병행.
