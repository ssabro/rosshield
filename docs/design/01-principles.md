# 01. 설계 원칙

본 제품의 모든 기술적 결정은 다음 **12개 원칙**에 뿌리를 둡니다. 원칙은 우선순위 순이며, 충돌 시 숫자가 작은 쪽이 이깁니다.

---

## P1. 감사인이 받아들이는 증거 (Auditable Evidence)

- 모든 결과는 **결정론적**이며 **재현 가능**해야 한다.
- 결과는 원본 명령 출력·설정 파일 스냅샷을 포함하는 **Evidence 레코드**와 분리 불가능하게 묶인다.
- 감사 추적은 **해시 체인**으로 연결되며, **외부 검증 API**로 제3자가 독립 검증할 수 있어야 한다.
- 리포트는 **기기 고유 키로 서명**되고 검증자가 원격 서명 확인을 수행할 수 있다.

**근거**: 상용 도구의 본질 가치는 "이 결과를 감사인 앞에 낼 수 있다"이다.

---

## P2. 옵트인 지능화 (Opt-in Intelligence)

- 모든 AI·LLM·통계 기능은 **기본 비활성**.
- 사용자가 명시적으로 활성화하지 않아도 핵심 기능(스캔·조치·리포트)은 **완전하게 동작**해야 한다.
- 활성화된 AI 기능도 **결정론적 fallback 경로**를 반드시 가진다.

**근거**: 보안 도구의 핵심은 예측 가능성. 의도하지 않은 외부 호출·데이터 흐름은 신뢰를 해친다. `nrobotcheck` v2에서 이미 검증된 원칙.

---

## P3. 에어갭 1급 (Air-gap First-class)

- 기본 배포 형태는 **오프라인에서 완전 동작**.
- LLM·업데이트·벤치마크 팩은 **오프라인 번들**(서명된 tar/zip)로 전달 가능해야 한다.
- 제품 내 어떤 경로도 인터넷 연결을 **"가정"**하지 않는다 — 있으면 쓰고, 없으면 대체 경로가 있어야 한다.

**근거**: 방산·병원·공항·제조 공정 OT 네트워크는 인터넷 접근이 원칙적으로 차단된다. SaaS-only로는 이 시장에 못 들어간다.

---

## P4. 테넌시 기본값 (Multi-tenancy by Default)

- **모든 테이블은 `tenant_id`를 가진다**. 단일 사용자 데스크톱 모드도 "기본 테넌트 1개"의 degenerate case.
- **모든 API는 테넌트 스코프 안에서 작동**. cross-tenant 참조는 감사 레이어를 거쳐야만 가능.
- 대형 고객용 **schema-per-tenant 옵션**을 설계 수준에서 미리 열어둔다.

**근거**: 단일 사용자 가정을 후행 리팩토링하는 비용은 처음부터 테넌시를 넣는 비용의 10배 이상. `nrobotcheck`가 이 리팩토링 부담을 회피한 게 신규 프로젝트의 주된 동기.

---

## P5. 도메인 주도 경계 (DDD Boundaries)

- `nrobotcheck` v2.0 도메인 분해를 승계·정제한다: `tenant`, `fleet`, `robot`, `benchmark`, `scan`, `evidence`, `insight`, `advisor`, `compliance`, `reporting`, `audit`, `platform`.
- 도메인 간 통신은 **이벤트 버스 + 명시적 API**로만. 도메인 서비스가 다른 도메인의 저장소를 직접 호출하지 않는다.
- 각 도메인은 자신의 **모델·저장소·서비스·정책·이벤트·API 스키마**를 소유.

**근거**: 다중 배포 타깃(데스크톱·서버·어플라이언스)에서 코드 재사용의 전제 조건.

---

## P6. 결정론적 fallback (Deterministic Fallback)

- AI 경로는 **언제나 규칙 기반 경로의 보조 수단**.
- LLM·외부 API 호출은 **timeout · 스키마 검증 실패 · 네트워크 장애** 전부를 rule-based로 흡수한다.
- 모든 LLM 응답은 **근거(reasoning trace) + 증거 ID + 신뢰도**를 포함 저장.

**근거**: 감사 도구가 LLM 장애로 멈추면 신뢰를 잃는다. `nrobotcheck`에서 `LlmRootCauseExplainer`·`ReportSummarizer`가 선례.

---

## P7. 단일 바이너리, 다중 껍질 (One Binary, Many Shells)

- 동일한 **서버 바이너리(또는 OCI 이미지)**가 데스크톱·온프렘·어플라이언스 3종 배포에 그대로 쓰인다.
- UI 코드베이스는 **단일 웹 앱**. 데스크톱 셸은 그 웹 앱을 호스팅하는 얇은 껍질(Tauri/Electron/WebView2).
- CLI는 같은 HTTP API의 또 다른 클라이언트.

**근거**: 3배 코드 중복은 제품 수명 중 반드시 품질 격차를 낳는다.

---

## P8. 컨텐츠는 코드와 분리 (Content / Code Separation)

- 벤치마크·컴플라이언스 매핑·리포트 템플릿·규제 프로필은 **서명된 "팩"**으로 배포.
- 제품 버전과 컨텐츠 버전은 **독립적**으로 릴리스된다.
- 팩은 오프라인 번들 설치 + (옵션) OTA 업데이트 양쪽 모두 지원.

**근거**: 규제 변경·벤치마크 갱신 주기는 제품 릴리스 주기보다 훨씬 빠르다. 컨텐츠 구독이 수익의 본체.

---

## P9. 데이터 불변성 (Immutable Data)

- 스캔 결과·Evidence·Insight·Audit 로그는 **append-only**. 수정·삭제 대신 새 버전 생성.
- 삭제가 필요한 경우(개인정보, GDPR)도 **tombstone + 교차 참조 유지**.

**근거**: 감사 추적·롤백·포렌식의 전제. 한 번 "통과"로 기록된 것이 나중에 수정 가능하면 감사 가치가 사라진다.

---

## P10. 프라이버시 기본값 (Privacy by Default)

- 로봇에서 수집한 원본 데이터는 **로컬 우선**.
- 외부 LLM·클라우드 전송 시 **사전 경고 + 민감 정보 레덕션(redaction)**.
- 기본 LLM provider는 **오프라인(Ollama/vLLM)**. 클라우드 전환은 명시적 사용자 선택.

**근거**: 고객사의 보안·개인정보 정책과 충돌하지 않기 위한 default.

---

## P11. 설명 가능성 (Explainability)

- 모든 AI 판단은 **근거(reasoning trace)**를 저장한다.
- 점수·위험도·조치 추천에는 "왜 그런가"를 **구조화된 필드**(`evidence_refs`, `rules_applied`, `confidence`)로 남긴다.
- UI는 **항상 근거를 펼쳐볼 수 있어야 한다**.

**근거**: "AI가 그렇다고 했다"를 근거로 보안 조치를 내릴 수 없다.

---

## P12. 점진적 적용 (Incremental Everything)

- 스키마 마이그레이션·API 버저닝·벤치마크 팩 업데이트·어플라이언스 OTA — 전부 **롤백 가능**하게 설계.
- **Big-bang 업그레이드 금지**. 모든 변경은 **N·N+1 공존 기간**을 거친다.
- 테스트: 각 변경은 **이전 버전 상태에서 새 버전으로 이전**하는 통합 테스트를 반드시 동반.

**근거**: 상용 제품은 고객 사이트에서 다운타임이 곧 해지 사유.

---

## 원칙 간 트레이드오프 처리

충돌이 생기면 **번호가 작은 원칙이 이긴다**.

**예시**:
- "AI 기능이 더 편하다" vs P2(옵트인) → P2 승리, 기본 비활성.
- "클라우드 연결로 LLM 쓰면 품질 좋아진다" vs P3(에어갭) → P3 승리, 오프라인 모드가 주력.
- "단일 테넌트 가정하면 스키마 단순해진다" vs P4(테넌시 기본값) → P4 승리, 처음부터 `tenant_id`.
- "빠른 구현을 위해 Electron main에 SSH 로직 직접 넣자" vs P7(단일 바이너리) → P7 승리, 코어는 Electron 의존 없이.

## 원칙 적용의 증거 (Verification)

각 원칙은 다음 방식으로 구현·검증된다:

| 원칙 | 검증 수단 |
|---|---|
| P1 | 서명 검증 통합 테스트, 외부 검증 API E2E |
| P2 | `AI_ENABLED=false` 전체 테스트 스위트 녹색 |
| P3 | 네트워크 차단 컨테이너에서 전체 E2E 녹색 |
| P4 | cross-tenant 접근 차단 테스트, SQL fuzzer |
| P5 | 도메인 간 import 린트 규칙, 계약 테스트 |
| P6 | LLM mock 실패 주입 테스트 |
| P7 | 같은 바이너리로 3종 배포 시나리오 스모크 테스트 |
| P8 | 팩 서명·설치·롤백 테스트 |
| P9 | 모든 UPDATE/DELETE 사용처 린트 규칙 |
| P10 | 레덕션 파이프라인 단위 테스트, 데이터 흐름 감사 |
| P11 | Insight/Recommendation JSON 스키마 필수 필드 검증 |
| P12 | 마이그레이션 왕복(up/down) 테스트 |

이 검증 수단들은 CI에 반드시 포함되어야 합니다. 원칙이 지켜지는지는 리뷰 의견이 아니라 **테스트로** 증명합니다.
